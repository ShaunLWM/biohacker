# Biohacker v1 Design

Date: 2026-04-14
Status: Approved for planning
Project root: `/Users/shaun/Desktop/Projects/biohacker`

## Summary

`biohacker` is a pnpm monorepo for an exploratory security-lab control plane.

V1 is intentionally narrow:

- A TanStack Start frontend shows a single control surface.
- A privileged Node daemon runs on the same Ubuntu 24.04 bare-metal host as Firecracker.
- Clicking `Create VM` boots one ephemeral Ubuntu microVM, injects SSH access, and returns SSH details to the frontend.
- Clicking `Shutdown` deletes the VM and all of its runtime artifacts.
- Every VM also has a configurable TTL and is force-cleaned when it expires, even if the user is still connected.

The system is designed for single-host operation now, but the daemon should keep a clean internal abstraction so remote worker hosts can be added later.

## Goals

- Create a pnpm monorepo with `apps/web` and `apps/daemon`.
- Scaffold the frontend with the official TanStack CLI and current supported add-ons.
- Run container-friendly services with Docker Compose on a bare-metal host.
- Run Firecracker from a host-level daemon, not from inside Docker.
- Boot Ubuntu-based Firecracker VMs with SSH access.
- Return SSH connection details immediately after VM creation.
- Enforce a configurable VM TTL.
- Delete instance disks, keys, sockets, tap devices, and forwarding rules on shutdown or expiry.

## Non-Goals

- No auth enforcement in v1, even if auth packages are scaffolded.
- No persistent VM catalog or resume flow.
- No multi-host scheduling in v1.
- No snapshots, save/restore, or persistent user disks.
- No public multi-tenant hardening in v1.

## Confirmed Product Decisions

- Monorepo name: `biohacker`
- Host scope: single host now, interfaces prepared for future multi-host expansion
- Guest OS: Ubuntu cloud image
- Access model for v1: no auth, exploratory only
- SSH reachability: host public IP plus forwarded host port
- Disk lifecycle: immutable base image plus disposable per-VM writable disk
- Shutdown behavior: delete instance immediately
- Host OS target: Ubuntu 24.04 LTS
- Frontend scaffold preference: use the TanStack CLI with Better Auth, Drizzle, TanStack Query, Biome, and Nitro
- Deployment shape: hybrid
  - Docker Compose owns containerized services
  - systemd owns the privileged Firecracker daemon
- TTL: configurable, example default `60` minutes

## Current Research Notes

### TanStack CLI

The current TanStack CLI supports the relevant deployment, toolchain, and add-on IDs:

- deployment: `nitro`
- toolchain: `biome`
- add-ons: `better-auth`, `drizzle`, `tanstack-query`

For this project, the Drizzle scaffold will use its default PostgreSQL option so the generated web stack has a concrete database target from day one, even though v1 VM state is not stored there.

It also reports that Tailwind is already enabled by default and `--tailwind` is deprecated.

The current `create` help exposes the relevant flags:

- `--target-dir`
- `--package-manager`
- `--deployment`
- `--toolchain`
- `--add-ons`
- `--non-interactive`

Planned scaffold shape during implementation:

```bash
npx @tanstack/cli create web \
  --target-dir apps/web \
  --package-manager pnpm \
  --deployment nitro \
  --toolchain biome \
  --add-ons better-auth,drizzle,tanstack-query \
  --non-interactive \
  --no-git
```

Note: the exact command will be smoke-tested during implementation against the live CLI before the repo scaffold is finalized, but the flags and IDs above were verified against the current CLI.

### Firecracker

Current Firecracker guidance confirms the following:

- Firecracker requires read/write access to `/dev/kvm`.
- Host-level TAP networking, IP forwarding, and NAT are the normal network setup path.
- Production usage should prefer `jailer` and restrictive seccomp defaults.
- Each Firecracker process should correspond to a single tenant workload.

This is why the design uses a hybrid deployment instead of putting the VM manager inside Docker Compose.

### Cloud-init / SSH Injection

The cleanest guest bootstrap for Ubuntu cloud images is a per-VM NoCloud seed image.

V1 will generate:

- `user-data`
- `meta-data`
- `network-config`

and build a local seed image with `cloud-localds`, then attach that seed as an additional drive so the guest boots with:

- an `ubuntu` user
- the generated SSH public key in `authorized_keys`
- a static guest IP configuration that matches the host TAP setup

This avoids needing a DHCP service on the host.

### Runtime Components

#### `apps/web`

TanStack Start frontend generated by the official CLI.

V1 responsibilities:

- Render a simple control page.
- Call the daemon API to create and shut down VMs.
- Show VM status, TTL, SSH host, SSH port, username, and private key text/download action.
- Recover active VM state on refresh using `GET /v1/vms`.

Even though Better Auth and Drizzle are scaffolded, v1 does not gate access on auth and does not depend on database-backed VM state.

#### `apps/daemon`

A host-level Node/TypeScript service exposing the VM control API and managing Firecracker lifecycle.

Suggested server shape:

- Fastify or equivalent minimal JSON HTTP framework
- explicit modules for:
  - config loading
  - VM allocation
  - Firecracker process control
  - host networking
  - disk preparation
  - cloud-init seed generation
  - TTL sweeping
  - reconciliation on daemon startup

#### `packages/shared`

Shared TypeScript contracts used by both apps:

- VM DTOs
- API request/response schemas
- config schema
- state enum definitions

#### `infra/` and `scripts/`

Host-facing operational assets:

- Firecracker download/install scripts
- Ubuntu base image preparation scripts
- networking helper scripts
- systemd unit files
- optional operator wrapper script to start both compose services and the daemon

## Deployment Model

### Hybrid Ownership

Docker Compose owns:

- `apps/web`
- PostgreSQL for the scaffolded Better Auth and Drizzle stack

The host owns:

- `biohacker-daemon.service`
- Firecracker binary
- jailer binary
- KVM access
- TAP devices
- host firewall/NAT rules
- VM state directories

### Operator Experience

Because the daemon must run on the host, `docker compose up -d` cannot honestly be the only startup primitive for the entire stack.

V1 should instead provide:

- `docker compose up -d` for all containerized services
- `systemctl enable --now biohacker-daemon` for the privileged daemon
- an operator helper script such as `scripts/up-host.sh` that runs both in the right order

This keeps the real runtime model honest while still giving you a near one-command deployment flow.

## Host Bootstrap Design

### One-Time Host Setup

`scripts/bootstrap-host.sh` will target Ubuntu 24.04 LTS and prepare:

- Firecracker and `jailer`
- a dedicated unprivileged service account for Firecracker workloads
- KVM access checks
- packages for image and seed preparation
  - `qemu-utils`
  - `cloud-image-utils`
  - networking tools like `iproute2`, `iptables` or nftables-compatible package set
- directory layout:
  - `/opt/biohacker/firecracker`
  - `/var/lib/biohacker/base-images`
  - `/var/lib/biohacker/instances`
  - `/var/log/biohacker`

The script will also:

- enable IPv4 forwarding
- detect the primary host interface
- prepare baseline NAT configuration
- install the daemon systemd unit

### Firecracker Asset Preparation

The host bootstrap also prepares:

- a Firecracker-compatible kernel image
- an Ubuntu base rootfs image stored as an immutable reference image

Implementation note:

- Prefer official Firecracker release binaries for `firecracker` and `jailer`
- Prefer an Ubuntu 24.04 cloud image as the guest base
- Convert the guest root disk into a Firecracker-friendly raw/ext4-backed image if the source artifact is not already suitable

## VM Lifecycle Design

### Create Flow

When the frontend sends `POST /v1/vms`, the daemon performs:

1. Allocate a VM ID.
2. Reserve a guest private IP.
3. Reserve a unique host SSH port from a configured range.
4. Create a per-VM working directory under `/var/lib/biohacker/instances/<vm-id>`.
5. Create a disposable writable root disk derived from the immutable Ubuntu base image.
6. Generate an SSH keypair for that VM.
7. Generate cloud-init `user-data`, `meta-data`, and `network-config`.
8. Build a NoCloud seed image with `cloud-localds`.
9. Create and configure a TAP device for the VM.
10. Add host NAT and TCP forwarding from `host-ip:ssh-port` to `guest-ip:22`.
11. Launch Firecracker through `jailer`.
12. Configure machine, kernel, drives, and network through the Firecracker API socket.
13. Start the VM.
14. Wait for SSH readiness or a bounded boot timeout.
15. Return VM details to the frontend.

### Returned VM Details

The create response includes:

- `id`
- `status`
- `host`
- `sshPort`
- `username`
- `privateKey`
- `createdAt`
- `expiresAt`

The username is `ubuntu`.

### Shutdown Flow

When the frontend sends `POST /v1/vms/:id/shutdown`, the daemon:

1. Marks the VM as `shutting_down`.
2. Attempts graceful guest shutdown.
3. Falls back to force kill after a timeout.
4. Removes forwarding rules.
5. Deletes the TAP device.
6. Removes API sockets, logs, cloud-init seed, SSH keys, and writable root disk.
7. Deletes the per-VM working directory.
8. Marks the instance as `deleted`.

No instance is restartable in v1.

### TTL Expiry

Each VM receives an `expiresAt` at creation time from a config value such as:

```env
VM_TTL_MINUTES=60
```

The daemon runs a background sweeper that:

- checks active instances at a fixed interval
- shuts down any expired instance
- marks the reason as `expired` instead of user-initiated

If a user is still connected over SSH at expiry time, the VM is still terminated.

## Networking Design

### V1 Network Model

Each VM gets:

- one TAP device on the host
- one private guest IP
- one forwarded host TCP port for SSH

The user never SSHes to the guest private IP directly. The frontend shows:

- host public IP
- forwarded host port
- username `ubuntu`
- private key

### Guest Networking

To avoid managing DHCP, the daemon supplies a cloud-init `network-config` file with static addressing for the guest.

The host sets:

- TAP device address
- host forwarding/NAT rules
- outbound masquerade via the main host interface

### Future Multi-Host Compatibility

The daemon should keep a clean internal interface such as `HostRunner` or `WorkerAdapter` so the current local-host implementation can later be swapped for remote workers without changing the web contract.

## API Design

### Endpoints

`POST /v1/vms`

- creates a VM

`GET /v1/vms`

- returns only currently active VMs and their public connection details

`POST /v1/vms/:id/shutdown`

- shuts down and deletes a VM

`GET /health`

- reports daemon readiness and prerequisite status

### VM States

- `creating`
- `running`
- `shutting_down`
- `deleted`
- `failed`

## Runtime State Strategy

V1 does not require a database-backed control plane.

The daemon keeps active state in memory and treats the instance directories on disk as recoverable host truth during startup reconciliation.

On daemon restart, it scans `/var/lib/biohacker/instances` and reconciles:

- orphaned Firecracker processes
- stale TAP devices
- stale forwarding rules
- partially created instance directories

Anything inconsistent is cleaned up conservatively.

## Capacity And Config

V1 should expose explicit config for:

- `VM_TTL_MINUTES`
- `MAX_ACTIVE_VMS`
- `SSH_PORT_RANGE_START`
- `SSH_PORT_RANGE_END`
- `HOST_PUBLIC_IP`
- `HOST_INTERFACE`
- `INSTANCE_BASE_DIR`
- `BASE_IMAGE_PATH`
- `KERNEL_IMAGE_PATH`
- `FIRECRACKER_BIN`
- `JAILER_BIN`

If `MAX_ACTIVE_VMS` is reached, create requests fail fast with a clear error that the UI can display.

## Error Handling

### Create Failures

If any step fails during creation, the daemon must execute the same cleanup plan it would use for shutdown and return a failure response without leaking resources.

### Firecracker Process Hangs

The daemon must supervise Firecracker child processes and kill unresponsive workloads.

This aligns with current Firecracker production guidance, which recommends an overwatcher process for unresponsive VMs.

### Log And Serial Handling

Firecracker serial and logs can grow due to guest activity.

V1 should either:

- disable guest serial output where practical, or
- route it into bounded storage handled by journald or rotated files

Unbounded stdout handling is not acceptable.

## Security Posture For V1

V1 is an exploratory control plane, not a public multi-tenant security platform yet.

That matters because your long-term goal involves users running security exercises and potentially hostile workloads. Before public exposure, at minimum the project will need:

- auth and authorization
- tenant isolation review
- outbound egress controls
- per-tenant limits
- abuse monitoring
- stronger network policy
- host hardening review
- secure secret handling

Even in v1, the daemon should still:

- use `jailer`
- avoid broad file permissions
- isolate per-VM files
- keep bounded logs
- enforce VM TTL and max concurrency

## Testing Strategy

### Unit Tests

- config validation
- IP and SSH port allocation
- TTL calculations
- cleanup planning
- API schema validation

### Integration Tests

- daemon create/shutdown lifecycle with mocked Firecracker process boundaries
- rollback behavior on partial failures
- startup reconciliation behavior

These tests should not require `/dev/kvm` in CI.

### Host Smoke Test

On a real Ubuntu 24.04 bare-metal machine:

1. Run host bootstrap.
2. Start compose services.
3. Start the daemon service.
4. Create a VM from the web UI.
5. SSH into it with the returned key.
6. Verify outbound network access if enabled.
7. Wait for TTL expiry.
8. Confirm the VM is terminated and resources are removed.
9. Repeat with manual shutdown.

## Known Deliberate Omissions

- No login requirement in v1
- No persistent instance history
- No snapshot support
- No browser terminal
- No per-user quota model
- No remote worker scheduling
